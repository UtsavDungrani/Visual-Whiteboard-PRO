Product Requirements Document
Defines what we are building, why we are building it, and what success looks like.

1.1  Project Overview
Product Name	Visual Whiteboard Pro
Project Type	UDP — User Defined Problem
Subject Code	BETXX17352 | Semester 7
Team Size	2 Members
Duration	1 Semester (~4 months)
Target Users	Software developers, system architects, students, tech teams

1.2  Problem Statement
Existing whiteboard tools like Excalidraw, Miro, and FigJam are generic drawing tools. They lack the intelligence to help developers design software architecture, and they do not support real-time team collaboration with AI-powered assistance. Developers waste time cleaning up messy diagrams, researching architecture patterns, and switching between multiple tools during system design sessions.
1.3  Proposed Solution
Visual Whiteboard Pro is a collaborative, real-time whiteboard application specifically designed for software architecture and system design. It combines canvas-based drawing with AI intelligence to help teams design, clean up, and validate software systems — all in one product.
1.4  Goals & Success Metrics
Goal	Success Metric	Target
Real-time collaboration	Multiple users editing simultaneously	< 200ms sync latency
Mess Cleanup feature	Auto-align drawn elements	< 2 seconds cleanup time
Architecture Assist	AI suggestions on diagrams	Relevant suggestion rate > 80%
Context Layer	Notes/URLs attached to elements	100% element coverage
Multi-page navigation	Smooth slide-like page switching	Page switch < 100ms
PDF Export	All pages exported accurately	Pixel-accurate at 150 DPI
HTML/CSS Export	Clean code from canvas elements	Valid HTML5 + CSS3 output

1.5  Core Features (In Scope)
Feature 1: Drawing Canvas
•	Shapes: rectangle, circle, diamond, cylinder, cloud
•	Connectors: arrows, lines, dashed connectors
•	Text labels, sticky notes, freehand drawing
•	Select, move, resize, delete, group elements
Feature 2: Real-Time Collaboration
•	Multiple users on same whiteboard simultaneously
•	Live cursor presence (see where others are pointing)
•	WebSocket-based sync via Redis state management
•	User color coding to identify participants
Feature 3: Mess Cleanup (AI)
•	Detect randomly placed shapes and auto-align to grid
•	Auto-route overlapping arrows cleanly
•	Apply consistent spacing and sizing to elements
•	One-click trigger with undo support
Feature 4: Architecture Assist (AI)
•	Analyze drawn system diagrams and detect component types
•	Suggest missing components (load balancer, cache, queue)
•	Recommend suitable databases (SQL vs NoSQL based on usage pattern)
•	Provide scalability notes and performance tips
Feature 5: Context Layer
•	Attach notes, URLs, code snippets to any element
•	Expand/collapse context panels on click
•	Support markdown in notes
•	Export context with diagram
Feature 6: Multi-Page System
•	Two modes: Infinite Canvas (freeform, no boundary) and Fixed Page Mode (A4 / 16:9 slide / custom size)
•	In Fixed Page Mode: visual page boundary shown on canvas, auto-prompt to add next page when content overflows boundary
•	Page panel (left sidebar): thumbnail strip showing all pages, click to navigate
•	Drag to reorder pages, duplicate page, delete page
•	Each page is an independent FabricJS canvas instance stored in the same whiteboard
•	Page counter and navigation arrows (Prev / Next) always visible at bottom
Feature 7: Export System
  7a. Export as PDF
•	Export all pages or selected pages as a single PDF
•	Each page maps to one PDF page, preserving canvas dimensions
•	Uses html2canvas to rasterize canvas + jsPDF to compile PDF
•	Options: page size (A4, 16:9), orientation, resolution (72/150/300 DPI)
  7b. Export as HTML/CSS
•	Export selected pages as HTML/CSS — user chooses format before export
•	Option A — Single File: all selected pages as <section> blocks in one HTML file with inline CSS
•	Option B — Separate Files: each selected page as its own self-contained .html file, bundled in a .zip download
•	Canvas elements converted to semantic HTML + CSS (absolute positioning)
•	Text elements → <p> / <h> tags, shapes → <div> with border-radius, images → <img>
•	Connectors/arrows → SVG inline within the HTML
•	Generated code is clean, readable, and copy-paste ready
1.6  Out of Scope (Not in this version)
•	Mobile app (iOS/Android)
•	Video/audio calling integration
•	Billing or subscription system
•	Offline mode
•	Export as React/Vue component (future scope)

3.2  Why Each Technology?
React
•	Component-based architecture maps perfectly to whiteboard elements (shapes, panels, toolbars)
•	Large ecosystem of supporting libraries
•	Team already familiar — reduces ramp-up time
FabricJS (over plain HTML Canvas)
•	Provides built-in object model for shapes, text, images on canvas
•	Handles selection, scaling, rotation automatically
•	Supports serialization to/from JSON — critical for saving and syncing state
•	Active community and well-documented API
WebSockets via Socket.io
•	Enables bi-directional real-time communication between clients
•	Automatic reconnection and fallback to long-polling
•	Room-based model fits per-whiteboard collaboration perfectly
Redis
•	Stores active whiteboard state in memory for ultra-fast reads
•	Pub/Sub model used to broadcast changes to all connected clients
•	Prevents data loss during WebSocket reconnects by caching latest state
MongoDB
•	Schema-flexible — whiteboard elements vary in structure (shape, text, connectors)
•	JSON-native storage aligns with FabricJS JSON serialization
•	Horizontal scaling built-in for future growth
Claude API for AI Features
•	Used for Mess Cleanup: send element positions as JSON, receive cleaned layout
•	Used for Architecture Assist: send diagram description, receive structured suggestions
•	Anthropic's API supports structured JSON output — easy to parse and apply
jsPDF + html2canvas
•	html2canvas renders the FabricJS canvas DOM node to a bitmap
•	jsPDF takes that bitmap and compiles one or multiple pages into a PDF
•	Both run entirely in the browser — no server-side processing needed
•	Supports multi-page PDF by iterating through each page canvas
JSZip
•	Bundles multiple HTML files into a single .zip for "Separate Files" export
•	Runs in-browser, no server upload required
•	Simple API: zip.file("page-1.html", htmlString) then zip.generateAsync("blob")
Custom HTML/CSS Serializer
•	FabricJS objects serialized to JSON, then mapped to HTML elements
•	IText/Text → <p> or <h1-h6> with inline font styles
•	Rect/Circle/Shape → <div> with border-radius, background, position CSS
•	Images → <img> with base64 src (self-contained)
•	Arrows/Lines → inline <svg> with <path> elements
•	All positions use CSS absolute within a relative container matching canvas size
Multer + Local Disk Storage (replaces AWS S3 / Cloudinary)
•	Multer is an Express middleware that handles multipart/form-data file uploads with zero cost
•	Uploaded files (images, context attachments) saved to /server/uploads/ directory on the host machine
•	On Render free tier: files persist within the same deployment session; for permanent storage, export/import JSON backups
•	File URLs served as static assets via Express: app.use("/uploads", express.static("uploads"))
•	For context layer file attachments: store filename + path in MongoDB, serve via /uploads/:filename route
•	Zero cost, zero configuration, no API keys — perfect for semester project scope
•	Future upgrade path: swap Multer destination to Cloudinary or S3 by only changing the storage engine, no other code changes

5.5  Redis Usage
•	Key: board:{boardId}:page:{pageId}:state → Stores FabricJS canvas JSON per page
•	Key: board:{boardId}:pages → Ordered list of page IDs
•	Key: board:{boardId}:users → Set of active user IDs
•	Key: board:{boardId}:userpages → Hash of userId → currently viewed pageId
•	Key: board:{boardId}:cursors:{pageId} → Hash of userId → {x, y} per page
•	TTL: 24 hours on all board keys (auto-cleaned if inactive)
•	On client connect → load all page states from Redis, sync to MongoDB every 30s
•	Pub/Sub channel: board:{boardId} → Used by Socket.io adapter for multi-server support
5.6  AI Service Design
Mess Cleanup Flow
1.	Receive array of element positions and types from canvas
2.	Format as structured JSON prompt for Claude API
3.	Claude returns cleaned layout: new x, y, width, height per element
4.	Backend validates response, returns delta to client
5.	Client animates elements to new positions
Architecture Assist Flow
6.	Client sends diagram summary: list of component types and connections
7.	Claude API prompted to act as a software architect reviewer
8.	Response: JSON array of {type, severity, message, suggestedComponent}
9.	Client renders suggestions in the Assist Panel
10.	User clicks "Apply" → component added to canvas via FabricJS

6.4  Risk Register
Risk	Probability	Impact	Mitigation
2-person team bandwidth	High	High	Strict phase-by-phase build; no feature added mid-sprint without removing another
WebSocket sync bugs with multiple users	Medium	High	Test with 2+ browser tabs early; use Redis locking
Claude API rate limits or cost	Low	Medium	Use free tier credits; cache AI responses; debounce calls
FabricJS version incompatibility	Low	High	Pin to v5.x; test all features before upgrading
html2canvas rendering mismatch vs canvas	Medium	Medium	Test PDF export early; use FabricJS toDataURL as fallback
HTML/CSS serializer edge cases	High	Low	Scope to basic elements only; document unsupported types
Render free tier sleep delay in demo	High	Medium	Ping server before presentation; keep backup localhost running
Local uploads/ lost on Render redeploy	Medium	Medium	For demo: only upload small test files; note limitation in presentation
